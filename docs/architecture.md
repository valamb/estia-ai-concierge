# ESTIA — System Architecture

## Overview

ESTIA is built as a layered, service-oriented application. Each layer has a single, well-defined responsibility. This makes the system easy to understand, test, and extend.

```
┌─────────────────────────────────────────────┐
│              Guest / Frontend               │
│         (Browser, Mobile, Kiosk)            │
└────────────────────┬────────────────────────┘
                     │ HTTP / WebSocket
┌────────────────────▼────────────────────────┐
│              FastAPI REST API               │
│           app/api/  (Route layer)           │
└────────────────────┬────────────────────────┘
                     │
┌────────────────────▼────────────────────────┐
│              Services Layer                 │
│  app/services/  (Business Logic)            │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │  Chat    │  │   RAG    │  │  Voice   │  │
│  │ Service  │  │ Service  │  │ Service  │  │
│  └────┬─────┘  └────┬─────┘  └──────────┘  │
└───────┼─────────────┼───────────────────────┘
        │             │
┌───────▼─────┐ ┌─────▼──────────────────────┐
│   OpenAI    │ │   ChromaDB Vector Store     │
│   GPT-4o    │ │   (RAG Knowledge Index)     │
└─────────────┘ └─────────────────────────────┘
                        ▲
              ┌─────────┴──────────┐
              │  knowledge/ folder │
              │  (Raw documents)   │
              └────────────────────┘
```

## Component Responsibilities

### `app/api/`
- Receives HTTP requests from clients
- Validates input using Pydantic models
- Delegates to the services layer
- Returns structured JSON responses
- No business logic lives here

### `app/core/`
- Loads and validates environment variables (`config.py`)
- Configures application logging (`logging.py`)
- Handles application startup and shutdown events

### `app/services/`
- `chat_service.py` — Manages conversation history and calls OpenAI
- `rag_service.py` — Queries ChromaDB to retrieve relevant hotel knowledge
- `language_service.py` — Detects language and applies language rules
- `voice_service.py` — (Future) Transcribes audio via Whisper

### `app/models/`
- Pydantic schemas for API requests and responses
- Ensures all data in and out is strictly typed and validated

### `app/utils/`
- Shared helper functions (text cleaning, formatting, etc.)
- Not specific to any single service

### `knowledge/`
- Raw hotel documents: Markdown, PDF, DOCX
- Organized by property and service category
- These documents are ingested into ChromaDB during setup

### `chroma_db/` (generated, not in git)
- The vector database created from `knowledge/` documents
- Stored locally; regenerated by running the ingestion script

## Data Flow — A Guest Question

```
1. Guest sends: "What time does the spa open?"
2. FastAPI receives the POST /chat request
3. ChatService is called with the message + conversation history
4. RAGService queries ChromaDB → returns top-5 relevant spa documents
5. ChatService builds a prompt: [system prompt] + [retrieved docs] + [conversation]
6. OpenAI GPT-4o generates a response
7. Response returned to guest via FastAPI
```

## Technology Choices

| Technology | Reason |
|---|---|
| **FastAPI** | Modern, async, auto-generates API docs (Swagger) |
| **OpenAI GPT-4o** | Best-in-class language understanding, multilingual |
| **ChromaDB** | Simple, local-first vector store, no external service needed |
| **LangChain** | RAG pipeline utilities (document loaders, splitters) |
| **Pydantic** | Data validation, environment configuration |
| **Loguru** | Simple, powerful logging |
